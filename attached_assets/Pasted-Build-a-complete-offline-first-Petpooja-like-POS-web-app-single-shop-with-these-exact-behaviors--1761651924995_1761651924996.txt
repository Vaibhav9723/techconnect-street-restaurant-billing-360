Build a complete offline-first Petpooja-like POS web app (single-shop) with these exact behaviors — IMPORTANT: a single fixed master password will be used and **must be** "0905". There is NO UI option for users to change this password.

PROJECT SUMMARY
- Tech: React (Vite) + TailwindCSS + React Router + Context API
- Offline-only: all data in localStorage, encrypted at rest
- Single shop per installation
- One-time unlock password (fixed): "0905"
- No server, no license, no user accounts

MASTER-PASSWORD RULES (ABSOLUTE)
1. The only accepted master password is `"0905"`. The app MUST accept `0905` when first unlocking on any device.
2. There will be NO user-facing option to change this password in the app. Do not provide "change password" UI.
3. When a device is unlocked by entering `0905`, the app will derive a key from that password and store a verifier locally so the device won't ask again on subsequent opens (until user clears browser storage).
4. If you (the owner) want to give access to someone, you tell them `0905`; when they open the website on their device and enter `0905`, the app unlocks on their device (and stores verifier locally).
5. Password must never be stored as plaintext. Only derived-key verifier + salt + kdf params are stored.

ENCRYPTION SPEC (MANDATORY)
- Use Web Crypto Subtle API: PBKDF2 with HMAC-SHA256, iterations = 150000 (or higher), salt length 16 bytes.
- Derived key: AES-GCM 256-bit.
- Export raw derived key bytes and store base64 as `pos_key_hash` (verifier).
- Store salt as `pos_salt` and iter count as `pos_kdf_iter`.
- All app data (products, categories, bills, settings, tokens) must be stored *encrypted* under localStorage keys: `pos_products`, `pos_categories`, `pos_bills`, `pos_settings`, `pos_tokens`. Each value = JSON `{ ct: base64Ciphertext, iv: base64IV, ts: ISO }`.
- Provide helper functions: `deriveKeyFromPassword(password, salt, iterations)`, `exportKeyToBase64(key)`, `encryptJSON(key, obj)`, `decryptJSON(key, blob)`, `bufToBase64`, `base64ToBuf`.

APP BEHAVIOR
1. Activation / Unlock Flow
   - On app load, check `pos_key_hash`. If not present:
     - Present an **Unlock Screen** with only one input: "Enter Master Password".
     - Accept only `0905` (but derive/verify programmatically — do not hardcode plaintext comparisons in storage).
     - After user enters `0905`, derive key, export base64, store `pos_key_hash`, pos_salt (if generating — but salt must be consistent; choose program design: use either fixed generated salt per device or generate salt and store). For consistency: generate new random salt on first unlock and use PBKDF2 with `0905` + generated salt to produce verifier; store salt so subsequent re-derivations match.
     - After successful unlock, decrypt (or create) app data stores (empty default stores if none) and proceed to app.
   - On subsequent visits on same device, skip unlock (because `pos_key_hash` exists). If the user clears browser data, they must enter `0905` again.
   - There is NO UI to change the master password; do not implement change flow.

2. App Data
   - Products: add/edit/delete (product = `{ id, name, price, categoryId, addCount }`)
   - Categories: add/edit/delete (category = `{ id, name }`)
   - Bills: store `{ id, dateISO, items, subtotal, discount, gst, total, token }`
   - Settings: `{ shopName, address, gstOn, gstPercent, tokenVisible (bool), printLayout: 'A4'|'58mm'|'80mm' }`
   - Tokens stored with daily reset `{ date:'YYYY-MM-DD', count:N }` (encrypted)

3. Billing UI
   - Desktop: left product grid, right cart
   - Mobile: bottom tabs with Products and Cart
   - Product card: fixed size; absolute-position add-count badge top-right showing `addCount` persistently; badge must not change card dimensions.
   - Add to cart increments product `addCount`.
   - Cart supports quantity edit, remove, discount, GST calculation, checkout -> create bill and store encrypted.

4. Dashboard
   - Show totals and stats; filters: Today, Yesterday, This Month, This Year, Custom Range
   - Clicking bill opens invoice modal/print preview

5. Settings
   - Shop Name, Address, GST on/off and percentage, Token Visible toggle, Print layout selector
   - Export Encrypted Backup: download AES-GCM-encrypted JSON file (ciphertext only)
   - Import Encrypted Backup: ask for master password (use current `0905`-derived key or prompt `0905` if needed) to decrypt and import (offer merge/overwrite)
   - DO NOT provide "change password" option or password-reset option
   - Provide "Wipe Data" option to clear all localStorage (document that this is irreversible)

6. Printing
   - Printable invoice templates for A4 / 58mm / 80mm
   - Include Shop Name, Address, GST, Token (if visible), Date/Time, items, totals
   - Provide Print and Download PDF (html2pdf or similar)

7. Storage Keys (localStorage)
   - `pos_salt` (base64)
   - `pos_kdf_iter` (number)
   - `pos_key_hash` (base64 verifier of exported key)
   - `pos_products` `{ ct, iv, ts }`
   - `pos_categories`
   - `pos_bills`
   - `pos_settings`
   - `pos_tokens`
   - `pos_activated` = "1" (set after first successful `0905` unlock)

8. Implementation Notes (for the generated code)
   - Use a central module `src/utils/crypto.js` containing the Web Crypto helpers and wrappers for encrypt/decrypt JSON.
   - Use `src/hooks/useEncryptedStorage.js` for reading/writing encrypted blobs (abstract get/set so components use it).
   - Keep the derived CryptoKey only in memory while app is open; do not store raw key anywhere.
   - For verifier: export raw derived key bytes to base64 and store as `pos_key_hash`. For subsequent unlocks, derive, export and compare base64 strings.
   - When generating salt on first unlock: store it in `pos_salt` so that `0905` on other devices will not match unless you want cross-device behavior. NOTE: To allow owner to simply type `0905` on any other device and unlock, use the following design choice:
       - **Design choice A (recommended for your use-case):** Use a **fixed, deterministic salt** known to all devices so that entering `0905` on any device derives the same key and unlocks (this enables your workflow: you go to another device, enter `0905` and app unlocks). For example, use a constant salt string hardcoded into the app (but not the raw password). This is less secure but matches your requirement.
       - **Design choice B (more secure):** Generate per-device random salt on first unlock. Then `0905` will only unlock that generating device (owner would need to export/import backup to other devices). **You requested cross-device unlock by entering 0905**, so implement **Design choice A**: use an app-embedded constant salt value (e.g., base64 `'fixed_salt_value_here'`) so `0905` on any device derives identical key.
   - The prompt MUST instruct the generator to implement **Design choice A** so that `0905` unlocks on any device without importing keys.
